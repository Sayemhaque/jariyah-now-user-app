NOTE on the two "-1" samples at tMs=1200 and tMs=2600:
These are NOT a bug. The span CSS uses `transition: color 120ms ease`
(VideoPreview.tsx line 473). When activeWord changes from word N to
word N+1, both spans' colors animate for 120 ms. During that window
neither span's computed color is *exactly* rgb(147, 51, 234) — it's
an interpolated value. The probe in the user-supplied script tests
for an exact color match, so it briefly returns -1 mid-transition.
This is purely a measurement artefact; the underlying activeWord
state is correct (verified by the surrounding samples which show
the expected word N before and word N+1 after).

NOTE on the "-1" samples from tMs=6200 onward: audio.currentTime
reached 6.031 s (the file duration), audio.ended fired, and the
onEnded handler in VideoPreview.tsx (line 190) explicitly sets
`setActiveWord(null)`. This is the intended "playback finished"
behaviour — highlight is cleared once the reciter stops.
