QUESTION 4 — Does the progress bar go 0% → 100% smoothly without resetting?
================================================================================
YES — by design.  From ExportModal.tsx:238 (comment) and :578-617 (code):

    // ONE unified 0–100% progress bar (no reset between phases).

The progress value is a single monotonic scalar:
    • 0.0  – 0.6   during WebM render        (phase: composing)
    • 0.6            setProgress(0.6) at handoff to conversion
    • 0.6 + p·0.4   during MP4 conversion    (p = 0 → 1)
    • 1.0            on completion (done or fallback)

`pct = Math.min(100, Math.max(0, Math.round(progress * 100)))` and the bar
uses `transition-[width] duration-300 ease-out`, so the fill grows
smoothly and never snaps backwards.

In THIS test run the entire pipeline finished in ~3 seconds (1 ayat,
6 s duration, 720p), so the bar went 0% → 100% very quickly. The first
screenshot caught it at 0% mid-"composing"; the second screenshot (+3s)
already showed "Video ready!" with the bar implicitly at 100%. No reset
was observed.

QUESTION 5 — Does the phase label change as processing progresses?
================================================================================
YES — wired correctly (ExportModal.tsx:585-594):

    setProcessingPhase('composing')    // during WebM render
    setProcessingPhase('uploading')    // at conversion start; also p < 0.5
    setProcessingPhase('encoding')     // 0.5 ≤ p < 0.95
    setProcessingPhase('finalizing')   // p ≥ 0.95

PHASE_LABELS map (ExportModal.tsx:245-262) is exactly as specified:
    composing   → "Composing frames"        / "Rendering the canvas, frame by frame"
    uploading   → "Preparing for conversion"/ "Sending to the Python + ffmpeg encoder"
    encoding    → "Encoding to MP4"         / "H.264 + AAC, optimized for every platform"
    finalizing  → "Finalizing"              / "Almost done — preparing your download"

The label re-mounts via `key={phase}` so the qv-fade-up animation
(350 ms ease-out) cross-fades on every change. The 3 dots next to the
label bounce continuously (qv-dot-bounce 1.4s).

CAVEAT for this run: because ffmpeg is NOT installed on the local dev
server (see Question 6), the MP4 conversion threw immediately and the
fallback path was taken — so in this particular session the label only
showed "Composing frames" before jumping to "Video ready!". On a server
with ffmpeg installed, all four phase labels would be visible.
